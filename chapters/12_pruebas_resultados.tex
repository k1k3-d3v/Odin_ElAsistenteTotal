\chapter{Pruebas y resultados}

\section{Estrategia de pruebas}
La validación se organizará por capas y por casos de uso. Primero se comprobará cada servicio de forma aislada; después se evaluarán flujos completos donde Odín reciba una entrada, la procese, consulte contexto, ejecute una acción y registre el resultado.

\section{Pruebas funcionales}

Como primera prueba de autoreparabilidad se validó el flujo de diagnóstico de Ollama. El servicio se encontraba inicialmente inactivo, se arrancó mediante systemd y se comprobó la disponibilidad de la API local consultando \texttt{/api/tags}. Posteriormente se ejecutó \texttt{odin\_autorepair.py --repair}; al detectar que Ollama estaba activo y la API respondía, la herramienta no realizó ninguna acción destructiva ni reinicios innecesarios.

También se validó el envío de informes a Telegram mediante \texttt{odin\_autorepair.py --daily}. El mensaje se generó reutilizando la configuración ya existente en los scripts de mantenimiento del servidor y sin versionar credenciales en el repositorio. Finalmente se probó el modo \texttt{--alert}, que detectó avisos vigentes relacionados con runners de Ollama, eventos AMDGPU y ocupación de disco, enviando alerta únicamente al cambiar la firma del problema.

La ingesta de memoria se validó ejecutando manualmente el mismo wrapper que utiliza cron. La ejecución cargó el modelo \texttt{mxbai-embed-large}, procesó 35 archivos nuevos o modificados y finalizó con código de salida 0. Después se comprobó Qdrant: la colección \texttt{memoria\_ia} quedó en estado \texttt{green} con 3554 puntos. Esta prueba confirma que la automatización horaria no solo apunta a una ruta válida, sino que ejecuta el flujo real de actualización de memoria. La prueba también permitió detectar y corregir un problema de permisos en la caché de ingesta, creada anteriormente por \texttt{root}.

La prueba también registró señales útiles para evaluar estabilidad: uso de GPU, porcentaje de VRAM asignada, temperaturas, estado de contenedores relevantes y ocupación de disco. En la ejecución observada, \texttt{rocm-smi} informó temperaturas en torno a 38--40 grados y ausencia de procesos KFD activos tras el arranque del servicio, mientras que Docker no presentaba contenedores críticos en estado problemático.

La verificación del cron permitió comprobar primero que las rutas antiguas ya no estaban presentes en la configuración activa. Posteriormente, durante la reorganización del almacenamiento, el cron se pausó por completo de forma reversible. Esta prueba valida una decisión operativa importante: cuando el sistema está cambiando de estructura, es preferible detener automatizaciones recurrentes antes que permitir que ejecuten ingestas, backups o reportes sobre rutas que pueden quedar obsoletas en mitad de una migración. Los logs previos se conservan como evidencia, pero la configuración activa queda en estado controlado.

Como prueba adicional de operación se migró la ubicación de la biblioteca de Immich hacia \texttt{/mnt/almacen}. La validación comparó tamaño y número de ficheros entre origen y destino, actualizó \texttt{UPLOAD\_LOCATION}, levantó el stack de Immich y comprobó tanto el estado de los contenedores como la respuesta de la API local. El resultado fue positivo: origen y destino quedaron con 202 GB y 77.666 ficheros, los contenedores quedaron en estado \texttt{healthy} y la API local respondió correctamente. Esta prueba no solo mide que el servicio arranque, sino que demuestra una mejora de mantenibilidad: los datos multimedia dejan de competir directamente con el espacio del sistema raíz.

También se revisó el estado de Docker Compose. Los stacks \texttt{odin-master}, \texttt{immich}, \texttt{audio}, \texttt{frigate}, \texttt{home-assistant-proxy}, \texttt{netdata} y \texttt{pocket-tts} aparecieron activos. Tras mover el stack principal al árbol de Odín, se comprobaron endpoints de Open WebUI, Qdrant, Nextcloud, n8n, Immich, Netdata y Stirling PDF. No se eliminaron volúmenes ni carpetas históricas, ya que podían contener datos persistentes, modelos o evidencias útiles de experimentación.

\subsection{Validación completa del despliegue Docker}
La validación de Docker se planteó en tres niveles: estado de Compose, estado de contenedores y comprobación funcional de endpoints. El primer nivel confirmó que todos los servicios activos pertenecían a un proyecto Compose y que ya no quedaban contenedores manuales sin etiquetar. Este punto es importante porque un contenedor creado con \texttt{docker run} puede seguir funcionando, pero queda fuera de la documentación reproducible del sistema. En Odín, la situación validada fue de 23 contenedores activos, 0 contenedores sin Compose, 0 contenedores en estado problemático y 7 proyectos Compose.

El segundo nivel revisó servicios visibles mediante peticiones HTTP locales. Open WebUI respondió en \texttt{/health}; Qdrant devolvió la colección \texttt{memoria\_ia}; Immich respondió en su endpoint de ping; n8n y Nextcloud respondieron a sus puertos internos; Netdata devolvió información de agente y Stirling PDF respondió con \texttt{401}, que en este caso es un resultado correcto porque indica servicio activo con autenticación. Esta distinción es importante: una prueba no debe considerar error cualquier código distinto de \texttt{200}; debe interpretarse según el comportamiento esperado de cada servicio.

El tercer nivel consistió en generar un estado agregable por Home Assistant mediante \texttt{odin\_ha\_status.py}. El exportador escribe \nolinkurl{/home/k1k3/odin/public/ha_status.json} y se publica con \texttt{odin-health-http.service}. El JSON incluye estado global, número de contenedores, contenedores sin Compose, contenedores no sanos, proyectos Compose, estado de endpoints, uso de disco, puntos de Qdrant y resumen de autoreparación. La última validación devolvió \texttt{overall: ok}, 23 contenedores, 0 sin Compose, 0 no sanos, 7 proyectos Compose y \texttt{netdata: 200}. Con esto, Home Assistant puede consumir una señal compacta para automatizaciones mientras Netdata queda disponible como panel humano.

Esta prueba enlaza directamente con los requisitos del proyecto: Odín debe ser capaz de hacer, pensar y avisar. Docker Compose aporta reproducibilidad; los healthchecks y endpoints aportan evidencia de funcionamiento; el JSON de estado permite automatización; Netdata aporta supervisión visual; y Telegram cubre avisos proactivos cuando aparece una anomalía. La monitorización no queda como una pieza decorativa, sino como parte del comportamiento del asistente.
