Como primera prueba de autoreparabilidad se validó el flujo de diagnóstico de Ollama. El servicio se encontraba inicialmente inactivo, se arrancó mediante systemd y se comprobó la disponibilidad de la API local consultando \texttt{/api/tags}. Posteriormente se ejecutó \texttt{odin\_autorepair.py --repair}; al detectar que Ollama estaba activo y la API respondía, la herramienta no realizó ninguna acción destructiva ni reinicios innecesarios.

En una prueba posterior, una transcripción de vídeo produjo un prompt de 4265 tokens y Ollama registró truncado porque Open WebUI cargaba el modelo con un contexto de 4096. Se comprobó que \texttt{ministral-3:14b} admite hasta 262144 tokens y se configuró Odín con 16384. La validación de \texttt{/api/ps} mostró \texttt{context\_length: 16384}, y una generación limpia respondió correctamente. Con esta configuración el modelo ocupa aproximadamente 12,7 GB.

Antes de cada modelo se descargaron los runners residentes; se fijaron 4096 tokens de contexto, temperatura cero y un mismo prompt que pedía no inventar el cifrado de un NVMe cuando el contexto no lo especificaba.

El Llama 3 básico alcanzó 100,37 tokens/s en caliente y fue el más rápido, pero también el único que completó una laguna con datos no proporcionados. Ministral generó 57,57 tokens/s y Qwen 63,10 tokens/s, ambos sin inventar. Por ello la selección no se basó únicamente en latencia: para memoria personal y ejecución de herramientas, la obediencia y la calibración resultan más importantes que unas décimas de segundo.

Como prueba de límite se cargó \texttt{lfm2:latest}, un modelo MoE cuantizado de 23,8B, con el contexto real de 16384. La carga tardó 9,39 s y Ollama reservó 15,02 GB. \texttt{rocm-smi} registró 15,45 de 17,10 GB de VRAM, GPU al 100\% y 63 grados en el sensor de unión. Aunque llegó a responder, dejó solo 1,65 GB libres, insuficientes para convivir con otra carga relevante. Este resultado explica los runners terminados y justifica descartar el modelo mayor para uso diario.

También se validó el envío de informes a Telegram mediante \texttt{odin\_autorepair.py --daily}. El mensaje se generó reutilizando la configuración ya existente en los scripts de mantenimiento del servidor y sin versionar credenciales en el repositorio. Finalmente se probó el modo \texttt{--alert}, que detectó avisos vigentes relacionados con runners de Ollama, eventos AMDGPU y ocupación de disco, enviando alerta únicamente al cambiar la firma del problema.

La ingesta de memoria se validó ejecutando manualmente el mismo wrapper que utiliza cron. La ejecución cargó el modelo \texttt{mxbai-embed-large}, procesó 35 archivos nuevos o modificados y finalizó con código de salida 0. Después se comprobó Qdrant: la colección \texttt{memoria\_ia} quedó en estado \texttt{green} con 3554 puntos. Esta prueba confirma que la automatización horaria no solo apunta a una ruta válida, sino que ejecuta el flujo real de actualización de memoria. La prueba también permitió detectar y corregir un problema de permisos en la caché de ingesta, creada anteriormente por \texttt{root}.

Las tools corregidas se probaron a través de la API de chat de Open WebUI, no solo llamando a sus dependencias. La transcripción de YouTube mostró como fuente \texttt{get\_youtube\_transcript} y devolvió texto real. Home Assistant localizó \texttt{vacuum.d10\_plus\_gen\_2} al consultar ``aspirador'', frente a las coincidencias irrelevantes de la versión inicial. Frigate llamó a \texttt{ultima\_persona}, recuperó una detección reconocida y devolvió una captura JPEG de 1280 por 720 a través del proxy con CORS. Mealie creó una receta con ingredientes, pasos e imagen y las recetas de prueba se eliminaron tras la validación.

El dashboard de Home Assistant se validó en tres niveles. La API confirmó que el panel \texttt{odin-panel} estaba registrado en modo \textit{storage}, visible en la barra lateral y con seis vistas. El exportador publicó las entidades de Odín para servicios, métricas agregadas y reservas de ampliación, incluyendo 17 servicios disponibles, 29 contenedores, cero contenedores problemáticos y 3565 puntos en Qdrant. La vista de salud añadió 25 entidades reales de Health Connect para actividad, sueño, constantes y composición corporal.

Tras la revisión final, la configuración alcanzó seis vistas y 33 secciones. Se recorrieron programáticamente todas las entidades utilizadas por tarjetas, listas, calendario, gráficos, cámaras y controles y se compararon con los estados expuestos por Home Assistant. El resultado fue de cero referencias ausentes.

La imagen panorámica se cargó en el almacenamiento de imágenes de Home Assistant y se verificó mediante una URL del mismo origen. También se comprobó que no quedara ninguna tarjeta \texttt{iframe}. Esta prueba elimina dos fallos observados: fondos inaccesibles desde algunos dispositivos y bloqueos del navegador al intentar conectar una sesión pública con recursos privados HTTP. El instalador actualizó el dashboard existente sin crear duplicados.

La prueba confirmó que Home Assistant recibe potencia instantánea y energía acumulada de ambos enchufes, conserva histórico y permite calcular costes. Los resultados detallados se recogen en las Tablas~\ref{tab:consumo-observado}--\ref{tab:sensibilidad-coste}.

La prueba también registró señales útiles para evaluar estabilidad: uso de GPU, porcentaje de VRAM asignada, temperaturas, estado de contenedores relevantes y ocupación de disco. En la ejecución observada, \texttt{rocm-smi} informó temperaturas en torno a 38--40 grados y ausencia de procesos KFD activos tras el arranque del servicio, mientras que Docker no presentaba contenedores críticos en estado problemático.

La verificación del cron permitió comprobar primero que las rutas antiguas ya no estaban presentes en la configuración activa. Durante la reorganización del almacenamiento, el cron se pausó de forma reversible. Tras identificar el NVMe por UUID, montar el destino y validar Restic, la tarea de backup se reactivó a las 04:00. Esta secuencia evita ejecutar automatizaciones sobre rutas transitorias y demuestra una activación controlada.

La prueba específica de backup comenzó con una auditoría del mecanismo histórico \texttt{tar.gz}, que permitió detectar exclusiones obsoletas y validar una restauración por hash. La solución definitiva se simplificó con Restic. El NVMe Crucial de 1 TB quedó identificado por UUID y montado en \texttt{/mnt/backup\_nvme}; el repositorio se inicializó cifrado y la clave de recuperación se copió fuera del servidor con permisos \texttt{0600}. El script genera un volcado consistente de MariaDB, copia la infraestructura y la biblioteca de Immich, aplica retención y ejecuta \texttt{restic check}. Finalmente se restauró un fichero en un directorio temporal y se comparó su hash con el original.

Como prueba adicional de operación se migró la ubicación de la biblioteca de Immich hacia \texttt{/mnt/almacen}. La validación comparó tamaño y número de ficheros entre origen y destino, actualizó \texttt{UPLOAD\_LOCATION}, levantó el stack de Immich y comprobó tanto el estado de los contenedores como la respuesta de la API local. El resultado fue positivo: origen y destino quedaron con 202 GB y 77.666 ficheros, los contenedores quedaron en estado \texttt{healthy} y la API local respondió correctamente. Esta prueba no solo mide que el servicio arranque, sino que demuestra una mejora de mantenibilidad: los datos multimedia dejan de competir directamente con el espacio del sistema raíz.

También se revisó el estado de Docker Compose. Los stacks \texttt{odin-master}, \texttt{immich}, \texttt{audio}, \texttt{frigate}, \texttt{home-assistant-proxy}, \texttt{netdata} y \texttt{pocket-tts} aparecieron activos. Tras mover el stack principal al árbol de Odín, se comprobaron endpoints de Open WebUI, Qdrant, Nextcloud, n8n, Immich, Netdata y Stirling PDF. No se eliminaron volúmenes ni carpetas históricas, ya que podían contener datos persistentes, modelos o evidencias útiles de experimentación.

\section{Resultados obtenidos}
Los resultados obtenidos muestran que Odín ha superado la fase de prototipo conceptual. El sistema integra conversación, memoria documental, fotos, domótica, cámaras, monitorización, acceso remoto, alertas y autoreparabilidad. La evidencia principal no es una única prueba aislada, sino la convivencia de múltiples servicios funcionando de forma coordinada.

En la capa de interacción, Open WebUI actúa como interfaz principal y las tools permiten ejecutar acciones reales. En la capa de memoria, Nextcloud y Qdrant permiten guardar y recuperar información. En la capa visual, Immich permite trabajar con fotos y Frigate con cámaras. En la capa domótica, Home Assistant y HA Voice resuelven control de dispositivos, temporizadores, música y presencia. En la capa operativa, Netdata, Telegram y el exportador JSON ofrecen visibilidad sobre el estado del sistema.

Las capturas incorporadas aportan evidencia visual del panel de Open WebUI, las tools, la búsqueda de fotos en Immich, la consulta de memoria, Mealie, el dashboard de Netdata, Home Assistant, Frigate, consumo eléctrico y las notificaciones proactivas. Junto con las pruebas de API y las tablas de validación, permiten comprobar el sistema desde la perspectiva técnica y desde la experiencia de uso.

\section{Discusion}
El proyecto confirma que un asistente personal privado es viable, pero también muestra que el coste principal no es solo computacional. El reto real está en la integración: cada servicio funciona de una forma distinta, requiere credenciales, puertos, persistencia, logs, actualizaciones y decisiones de seguridad. Odín funciona porque no intenta resolver todo con un único modelo, sino combinando herramientas especializadas.

Lo que mejor ha funcionado es la arquitectura modular. Docker Compose permite aislar servicios, Home Assistant aporta una base domótica estable, Open WebUI ofrece interfaz y extensibilidad, Qdrant habilita memoria semántica, Immich cubre recuerdos visuales y n8n facilita automatizaciones. También ha funcionado bien la idea de monitorización proactiva: Telegram, Netdata y el exportador para Home Assistant convierten el mantenimiento en parte del propio asistente.

Lo más difícil ha sido la voz y la aceleración con AMD. La búsqueda de un TTS realista, rápido y compatible con ROCm consumió mucho esfuerzo y no produjo una solución perfecta. Las medidas confirman que Piper sí cumple la latencia interactiva, aunque no sea la voz más natural. En backup, el montaje persistente, el cifrado del repositorio, la copia de datos masivos y la restauración quedan validados; como mejora futura permanece una segunda copia física o remota para aproximarse a una política 3-2-1.

La discusión final es que Odín no debe evaluarse como una aplicación cerrada, sino como una infraestructura personal integrada. Su valor está en que permite hacer cosas reales: guardar, buscar, recordar, controlar, avisar, monitorizar y mantener. Precisamente por eso la memoria debe mostrar no solo el resultado, sino las decisiones que lo hacen sostenible.
