\chapter{Desarrollo de la propuesta y trabajo realizado}

\section{Preparacion del entorno}
El entorno de Odín se construye sobre un servidor doméstico con Ubuntu 24.04.4 LTS, procesador AMD Ryzen 5 5600G, GPU AMD Radeon RX 9070/9070 XT/GRE, 30 GiB de RAM y almacenamiento local dividido entre el disco principal y \nolinkurl{/mnt/almacen}. La elección de un servidor propio responde al objetivo central del proyecto: ejecutar servicios personales y de IA bajo control del usuario, sin depender de una plataforma cerrada.

La estructura operativa se reorganizó bajo \nolinkurl{/home/k1k3/odin}. Dentro de ese árbol se separan dominios: \texttt{core} para el núcleo conversacional y servicios base, \texttt{audio} para voz, \texttt{media} para Immich, \texttt{automation} para Home Assistant/Frigate y automatizaciones, \texttt{monitoring} para Netdata y exportadores, \texttt{scripts} para cron y mantenimiento, y \texttt{data} para persistencia de Open WebUI. Esta estructura no existió desde el primer día; surgió de ordenar un sistema que había crecido mediante pruebas y servicios añadidos progresivamente.

Docker Compose se usa como mecanismo principal de despliegue porque permite describir servicios, redes, volúmenes, reinicios y dependencias en ficheros versionables. La inspección operativa se realiza mediante comandos Docker, healthchecks, logs y el panel agregado de Home Assistant. En paralelo, algunos componentes se mantienen fuera de Docker por decisión de diseño: Ollama funciona como servicio systemd en el servidor, mientras que Tailscale y Home Assistant viven en una Raspberry Pi 5 independiente. Esta combinación evita forzar todo dentro de contenedores cuando una pieza ya es más estable o segura fuera de ellos.

\section{Fase 1: infraestructura base}
La infraestructura de cómputo parte de un servidor principal. Sobre él se instalaron Docker, Docker Compose, herramientas de monitorización, ROCm para la GPU AMD y servicios systemd auxiliares. El diseño evita exposición pública directa: el acceso remoto entra por Tailscale en la Raspberry Pi 5 y continúa hacia Odín mediante la LAN. El servidor no necesita ejecutar el nodo VPN.

El almacenamiento se dividió en dos zonas. El disco raíz aloja sistema, contenedores, configuraciones y servicios. \nolinkurl{/mnt/almacen} se reserva para datos pesados, especialmente la biblioteca de Immich. Esta separación se volvió necesaria al comprobar el crecimiento de fotos y vídeos. Como política de seguridad, no se eliminaron volúmenes Docker ni carpetas históricas sin inventario previo.

La estrategia final de backup utiliza un NVMe Crucial de 1 TB, identificado por UUID y montado de forma persistente en \nolinkurl{/mnt/backup_nvme} con permisos restrictivos. Sobre él se inicializó un repositorio Restic: cada copia queda cifrada, es incremental y aprovecha deduplicación, por lo que una única política puede cubrir tanto la infraestructura como los datos masivos sin duplicar innecesariamente bloques.

El script \nolinkurl{backup_diario.sh} genera primero un volcado consistente de MariaDB para Nextcloud y después copia \nolinkurl{/home/k1k3/odin} y la biblioteca de Immich situada en \nolinkurl{/mnt/almacen/immich}. Se excluyen cachés regenerables, logs, entornos virtuales, dependencias descargables y el propio fichero de apertura del repositorio. La retención conserva siete copias diarias, cuatro semanales y seis mensuales. Cada ejecución finaliza con \texttt{restic check}; la clave de recuperación se guarda con permisos \texttt{0600} fuera del servidor.

La tarea se ejecuta desde el \texttt{crontab} de \texttt{root}, porque algunos volúmenes Docker solo son legibles con privilegios administrativos. Si el NVMe no está montado o falta la clave local, el script termina con error antes de crear una copia incompleta. La primera ejecución procesó 212,133 GiB y creó el snapshot \texttt{45ac7833}. Las siguientes ejecuciones solo incorporan bloques nuevos o modificados. La política de retención mantiene siete estados diarios, cuatro semanales y seis mensuales; después se eliminan los bloques que dejan de estar referenciados. La comprobación lee metadatos y una muestra del contenido cifrado, y la restauración de prueba confirmó mediante SHA-256 que el fichero recuperado era idéntico al original.

\section{Fase 2: inferencia local}
La inferencia local se apoya principalmente en Ollama como servicio del sistema. Ollama permite servir modelos LLM mediante una API local y simplifica la gestión de modelos respecto a un despliegue manual de \texttt{llama.cpp}. Aun así, la base técnica de ejecución de modelos cuantizados y acelerados por hardware se relaciona con el ecosistema de \texttt{llama.cpp}, que se mantiene como referencia técnica para inferencia local eficiente \citep{ollamaDocs,llamacppRepo}.

El principal condicionante ha sido la GPU AMD. ROCm permite usar aceleración, pero su compatibilidad práctica no siempre es equivalente a CUDA. Durante las pruebas aparecieron avisos de \texttt{rocblaslt}, referencias a \texttt{gfx1200}, runners terminados con \texttt{signal: killed}, prompts truncados por ventana de contexto y eventos de evacuación de colas AMDGPU. Estas incidencias obligaron a tratar la inferencia como una carga que debe monitorizarse, no como un servicio que simplemente se instala y se olvida.

Por ese motivo, Odín combina tres decisiones. Primero, Ollama se mantiene como servicio systemd para poder arrancarlo, comprobarlo y repararlo con herramientas del sistema. Segundo, la autoreparabilidad no reinicia servicios de forma agresiva, sino que diagnostica antes de actuar. Tercero, algunas cargas se desplazan a CPU cuando usar GPU aporta más riesgo que beneficio, como ocurre con Frigate. En cambio, Immich sí usa ROCm porque su servicio de machine learning se beneficia de la aceleración sin saturar el flujo conversacional principal.

\section{Fase 3: orquestación y automatización}
La orquestación de Odín es deliberadamente híbrida. No se fuerza toda la lógica dentro de n8n ni se resuelve todo mediante código. La herramienta se elige según el tipo de problema. n8n resulta adecuado para flujos visuales que conectan webhooks, APIs, mensajería y servicios externos, especialmente cuando interesa inspeccionar entradas y salidas paso a paso. Los scripts Python y shell se reservan para procesos con lógica técnica propia, tratamiento intensivo de datos o acceso al sistema, como la ingesta documental, los informes de salud, la autoreparación y el backup. Cron programa tareas periódicas y systemd mantiene los servicios persistentes, controla sus reinicios y expone sus logs. Home Assistant conserva las automatizaciones domésticas, de forma que luces, presencia, voz y dispositivos sigan funcionando aunque el servidor de IA esté en mantenimiento.

Estas capas se complementan. Open WebUI recibe la intención del usuario y selecciona una herramienta; esa herramienta puede llamar directamente a una API, activar un webhook de n8n o delegar una acción en Home Assistant. A su vez, los scripts de mantenimiento trabajan en segundo plano sin depender de una conversación. Docker Compose describe y reproduce los servicios, pero no sustituye a la lógica de orquestación. Esta separación evita convertir una sola plataforma en un punto único de fallo y permite usar en cada caso el mecanismo más sencillo, observable y mantenible.

Durante la revisión operativa se detectó una situación típica de proyectos personales que crecen por iteración: existían scripts reorganizados dentro de \nolinkurl{/home/k1k3/odin/scripts/}, pero el cron de \texttt{root} seguía apuntando a rutas antiguas en \nolinkurl{/home/k1k3}. Esto producía errores periódicos aunque el sistema principal siguiera funcionando. La solución fue consolidar la programación bajo el usuario \texttt{k1k3}, desactivar el cron de \texttt{root} y dejar copias históricas antes de cada cambio.

La ingesta activa queda ejecutada por \nolinkurl{cron_odin_ingesta.sh}, que llama a \nolinkurl{odin_ingesta_master.py}. Este flujo indexa documentos de Nextcloud en Qdrant usando el modelo \nolinkurl{mixedbread-ai/mxbai-embed-large-v1}. Se evita activar simultáneamente el script incremental antiguo \nolinkurl{update_odin.py}, porque utiliza otro modelo de embeddings y podría introducir inconsistencias dimensionales en la colección vectorial. Esta decisión refleja una idea importante del proyecto: automatizar no significa ejecutar todo lo disponible, sino elegir un camino mantenible y coherente. Durante la reorganización de carpetas se ajustó la ruta de la bóveda de Nextcloud al nuevo árbol \nolinkurl{/home/k1k3/odin/core/odin-master} y se corrigieron permisos de la caché de ingesta para que cron pudiera actualizarla sin privilegios de \texttt{root}.

También se separó la configuración de Telegram en un fichero \texttt{telegram.env} con permisos restrictivos. De esta forma los scripts pueden enviar avisos sin dejar tokens incrustados en el código operativo ni en la memoria del proyecto.

\section{Fase 4: domótica e interacción}
Home Assistant es la capa domótica de Odín y se ejecuta en una Raspberry Pi 5 separada del servidor principal. Esta separación es deliberada: la domótica debe seguir funcionando aunque el servidor de IA esté reiniciándose, probando modelos o reorganizando contenedores. Odín se integra con Home Assistant mediante tools de Open WebUI, HA Assist, HA Voice y un proxy local.

Antes de consolidar esta arquitectura se evaluaron dos despliegues alternativos: Home Assistant en contenedor dentro del servidor y Home Assistant OS en una máquina virtual. El contenedor simplificaba la infraestructura, pero complicaba complementos, descubrimiento de red y mantenimiento de integraciones domésticas. La máquina virtual ofrecía mayor aislamiento, aunque añadía consumo y otra capa de administración. Tras estas pruebas se mantuvo Home Assistant OS en la Raspberry Pi, mientras que el servidor Odín asumió IA, automatización pesada, almacenamiento y servicios multimedia.

La interacción doméstica cubre luces, sensores, presencia, temporizadores, música, calendario, lista de la compra y consultas de estado. Los temporizadores y la música se gestionan desde HA Voice, que ya funciona correctamente y resulta más natural que duplicar esa lógica dentro de Open WebUI. También existe una automatización de Home Assistant que notifica cada vez que alguien llega, usando un sensor de presencia.

Desde el punto de vista del asistente, Home Assistant aporta dos niveles. El primero es estructurado: buscar entidades, elegir la correcta y ejecutar un servicio. El segundo es conversacional: enviar una orden natural a Home Assistant Assist y dejar que HA resuelva entidades, áreas e intents. Esta doble vía permite combinar control fino con experiencia natural.

\section{Fase 4.1: voz, STT y TTS}
La capa de voz ha sido una de las partes más complejas del proyecto. La entrada de voz requiere convertir audio en texto de forma fiable (STT/ASR), mientras que la salida exige generar una voz suficientemente natural y rápida (TTS). En un asistente cotidiano, la calidad de esta capa determina si el sistema se siente vivo o torpe: una respuesta técnicamente correcta pierde valor si tarda demasiado o si la voz resulta incómoda.

Para STT se han probado varias aproximaciones, incluyendo Whisper, faster-whisper, Wyoming Whisper y un servicio ASR propio basado en ONNX. Esta diversidad responde a la necesidad de equilibrar latencia, precisión, consumo y facilidad de integración con Home Assistant y Open WebUI. En el estado actual conviven servicios de voz orientados a distintas pruebas: \texttt{faster-whisper-server}, \texttt{wyoming-whisper} y \texttt{odin-asr}.

La síntesis de voz ha sido todavía más problemática. El objetivo era conseguir una voz realista y rápida, pero la GPU AMD ha limitado muchas opciones por falta de soporte ROCm maduro. Se probaron alternativas como Piper, Kokoro, Chatterbox y PocketTTS. Piper se mantiene como opción estable para el asistente físico o móvil: no es la voz más realista, pero responde rápido y se integra bien. Chatterbox fue una de las pocas opciones que llegó a funcionar sobre GPU, pero su latencia resultó demasiado alta para interacción conversacional. PocketTTS quedó como línea actual de experimentación para una interfaz compatible con estilo OpenAI.

\section{Fase 5: nube privada y memoria}
La memoria de Odín se apoya en tres pilares: Nextcloud, Qdrant e ingesta automatizada. Nextcloud almacena documentos, notas, ideas, archivos personales y contenido que el usuario quiere conservar. Qdrant almacena vectores y metadatos para recuperación semántica. La ingesta convierte documentos en fragmentos, calcula embeddings y actualiza la colección \texttt{memoria\_ia}.

El objetivo no es únicamente guardar ficheros, sino poder preguntar por ellos. La herramienta \texttt{buscar\_nextcloud} obliga al modelo a consultar la memoria privada cuando la pregunta puede estar en notas o documentos. Esto reduce respuestas inventadas y permite distinguir entre conocimiento general y evidencia personal. A su vez, \texttt{guardar\_nota} permite añadir nuevo contenido desde la conversación, cerrando el ciclo entre crear, almacenar, indexar y recuperar.

La parte visual de la memoria se cubre con Immich. Las fotos se suben automáticamente y la tool \texttt{buscar\_fotos} permite pedir recuerdos en lenguaje natural. Así, la memoria de Odín no queda limitada a texto: combina documentos, notas, fotografías, vídeos, estados de sensores y eventos de cámaras.

\section{Fase 6: seguridad, acceso remoto y monitorización}
Dentro de esta fase se incorpora una primera herramienta de autoreparabilidad para Odín. El objetivo no es reiniciar servicios de forma indiscriminada, sino diagnosticar el estado del ecosistema y actuar solo cuando la reparación es segura. La herramienta \texttt{odin\_autorepair.py} comprueba el estado de \texttt{ollama.service}, la disponibilidad de la API local de Ollama, eventos recientes en \texttt{journalctl}, señales de presión en AMDGPU, estado de contenedores principales, métricas de GPU mediante \texttt{rocm-smi} y uso de disco.

La política de reparación es conservadora: por defecto solo genera un informe; con el modo \texttt{--repair} arranca Ollama si el servicio está inactivo; y solo reinicia un Ollama activo si se solicita explícitamente. Esta decisión evita que la capa de autoreparación introduzca más riesgo que el fallo original, especialmente en un sistema donde conviven contenedores, GPU compartida y servicios experimentales.

El primer caso de uso real de esta herramienta fue el diagnóstico de una caída parcial de Ollama. Los logs mostraban runners de modelo terminados con \texttt{signal: killed}, avisos de ROCm/\texttt{rocblaslt} relacionados con \texttt{gfx1200}, prompts truncados por superar la ventana de contexto y múltiples eventos de evacuación de colas AMDGPU. El servicio principal, sin embargo, había quedado parado de forma limpia por systemd. Tras arrancarlo de nuevo, la API local respondió correctamente y la herramienta confirmó que no era necesario ejecutar más acciones.

Como ampliación, la herramienta se integró con Telegram para ofrecer dos niveles de comunicación: un informe diario exhaustivo y alertas cuando aparece o cambia un problema relevante. El informe incluye el estado de Ollama, GPU, contenedores, disco, cron, Qdrant y logs de ingesta; además, enumera las fuentes revisadas y las fuentes de memoria procesadas recientemente. Esta integración convierte la monitorización en una funcionalidad proactiva real: Odín no solo almacena conocimiento, sino que informa sobre su propio estado operativo.

Durante esta integración se detectó una deuda técnica importante: el cron seguía apuntando a rutas antiguas de scripts que habían sido reorganizados. En una primera iteración se corrigieron las rutas hacia \texttt{/home/k1k3/odin/scripts/} y se definió una programación periódica para sincronización incremental, ingesta horaria, alertas e informe diario. Durante la reorganización se pausaron las tareas de forma reversible. Una vez montado y validado el NVMe, el backup quedó programado a las 04:00 con privilegios suficientes para leer los volúmenes persistentes de Docker.

\section{Fase 7: reorganización de almacenamiento}
El crecimiento de servicios como Immich obliga a separar con claridad el sistema operativo de los datos pesados. En la inspección del servidor se observó que el disco raíz concentraba una parte importante de la carga, mientras que \texttt{/mnt/almacen} disponía de espacio suficiente para actuar como almacenamiento auxiliar. Por ello se migró la biblioteca de Immich desde \texttt{/home/k1k3/Pictures/immich} hacia \texttt{/mnt/almacen/immich}.

La migración se realizó de forma conservadora: primero se detuvo únicamente el stack de Immich, después se guardó una copia de su fichero \texttt{.env}, se copiaron los datos con \texttt{rsync -aH} y, al terminar la copia, se modificó la variable \texttt{UPLOAD\_LOCATION}. Después se levantó de nuevo el stack y se verificó que \texttt{immich\_server}, \texttt{immich\_postgres}, \texttt{immich\_redis} e \texttt{immich\_machine\_learning} quedaban en estado \texttt{healthy}. El origen original se conserva temporalmente como respaldo antes de decidir su borrado definitivo.

Esta fase también define una política de limpieza prudente. Se pueden eliminar contenedores detenidos, imágenes Docker no utilizadas y caché de construcción, pero se evita borrar volúmenes Docker, modelos, bases de datos o carpetas de cuarentena sin una revisión explícita. En un proyecto como Odín, parte del valor académico está precisamente en los experimentos fallidos; limpiarlos sin inventario supondría perder trazabilidad sobre decisiones técnicas.

Los ficheros Compose constituyen la fuente reproducible y el inventario se valida mediante etiquetas de proyecto, estados de contenedor, healthchecks y endpoints funcionales.

\subsection{Stack \texttt{odin-master}: núcleo conversacional, datos y automatización}
El stack \texttt{odin-master} concentra once servicios y es el corazón histórico de Odín. No es la organización ideal a largo plazo, porque mezcla interfaz, automatización, almacenamiento, bases de datos, conectividad y utilidades. Sin embargo, su existencia es coherente con la evolución del proyecto: Odín no nació como un producto cerrado, sino como un sistema incremental en el que cada necesidad real iba añadiendo una pieza.

El contenedor \texttt{webui-odin} proporciona la interfaz principal. Open WebUI es especialmente relevante porque no se limita a ser una pantalla de chat: permite conectar modelos, usar herramientas, ampliar comportamiento mediante funciones y operar como una capa de interacción entre el usuario y el resto de servicios \citep{openwebuiDocs,openwebuiToolsDocs,openwebuiFunctionsDocs}. En Odín esta interfaz cumple tres funciones: conversación con modelos, acceso a herramientas propias y punto de entrada cotidiano para consultar memoria, fotos, domótica o estado del sistema.

\texttt{qdrant-odin} sostiene la memoria semántica. Qdrant permite almacenar vectores, colecciones y metadatos, lo que encaja con una arquitectura RAG en la que los documentos no se consultan únicamente por nombre de archivo, sino por similitud semántica \citep{qdrantDocs,qdrantStorageDocs}. En la práctica, la colección \texttt{memoria\_ia} actúa como índice de conocimiento personal: documentos de Nextcloud, notas y contenido extraído pueden convertirse en embeddings y recuperarse posteriormente como contexto.

\texttt{n8n-odin} representa una de las capas de orquestación, no la única. Su papel no es sustituir al código propio, sino reducir fricción al conectar llamadas HTTP, webhooks, calendarios, mensajes e integraciones entre servicios \citep{n8nDocs,n8nDockerDocs}. Cuando el proceso requiere manipulación compleja, acceso al sistema, pruebas automatizadas o control detallado de errores, se utiliza un script Python o shell. Cuando depende de sensores y dispositivos domésticos, se mantiene en Home Assistant. Esta distribución permite que una intención conversacional termine en una acción concreta sin concentrar toda la lógica de Odín en una sola herramienta.

\texttt{nextcloud-odin} y \texttt{nextcloud-db} aportan la nube privada. Nextcloud permite mantener documentos, notas, ficheros y material de trabajo en infraestructura propia \citep{nextcloudDocs}. La base \texttt{nextcloud-db}, basada en MariaDB, queda separada porque el estado de una nube privada es crítico: perder la base de datos puede ser más grave que perder una caché. Esta decisión conecta con la filosofía \textit{local-first}: los datos personales deben seguir estando bajo control del usuario aunque el sistema use capas de IA encima.

La pareja \texttt{evolution-odin} y \texttt{evolution-db} añade una vía de integración de mensajería. En un asistente proactivo, los canales de salida importan casi tanto como los de entrada: si Odín detecta un fallo, una tarea pendiente o un evento relevante, necesita poder avisar. Evolution API cubre ese espacio de integración con mensajería, mientras PostgreSQL mantiene su estado persistente \citep{evolutionApiDocs,postgres15Docs}. En la memoria se debe tratar como una capacidad de comunicación, no como una pieza aislada.

\texttt{redis-odin} aparece como infraestructura auxiliar. Redis se usa habitualmente como caché, cola o almacenamiento temporal de baja latencia \citep{redisDocs}. En un sistema con tantos servicios, disponer de un componente de este tipo evita que todas las operaciones dependan de bases de datos relacionales o ficheros. No guarda la memoria principal del proyecto, pero sí ayuda a que flujos y servicios respondan con menos latencia.

\texttt{cloudflared-odin} es una pieza histórica de conectividad. Cloudflare Tunnel permite conectar servicios internos con la red de Cloudflare sin abrir directamente puertos entrantes tradicionales \citep{cloudflareTunnelDocs}. Aun así, la decisión actual de Odín se inclina hacia Tailscale porque resuelve de forma más simple el acceso privado entre dispositivos, evita depender de DNS dinámico externo y encaja mejor con el uso por VPN descrito en la \cref{tab:vpn-comparativa} \citep{tailscaleWhatIs,tailscaleDynamicIp}. En la memoria conviene explicar esta convivencia como transición técnica, no como contradicción.

\texttt{crawl4ai} cubre una necesidad muy propia de Odín: transformar contenido web en material guardable, resumible e indexable. Su valor no está solo en descargar HTML, sino en servir de puente entre páginas, transcripciones y memoria \citep{crawl4aiDocs}. Esto conecta con herramientas de Open WebUI como guardar una página o una transcripción para que después se pueda buscar en Qdrant o archivar en Nextcloud.

\texttt{stirling-pdf} añade una utilidad local para manipular PDF \citep{stirlingPdfDocs}. Aunque pueda parecer secundaria, es coherente con la idea del asistente total: muchos documentos académicos, administrativos o técnicos llegan como PDF. Tener una herramienta local reduce dependencia de webs externas, evita subir documentos privados a servicios de terceros y mantiene la operación dentro de la infraestructura propia.

\subsection{Stack \texttt{immich}: fotos, recuerdos y almacenamiento pesado}
Immich representa la dimensión visual y personal de Odín. En lugar de tratar las fotos como simples archivos, el sistema las convierte en un repositorio consultable y automatizable. El contenedor \texttt{immich\_server} gestiona la aplicación principal, subida desde móvil, organización y API. La documentación oficial recomienda Docker Compose como método de despliegue, con un fichero \texttt{.env} que define, entre otros valores, la ubicación de la biblioteca \citep{immichDockerInstall}.

La decisión operativa más importante fue mover \texttt{UPLOAD\_LOCATION} a \nolinkurl{/mnt/almacen/immich}. Esto separa datos multimedia del sistema raíz. En la validación se comprobó que origen y destino tenían 202 GB y 77.666 ficheros, y que los contenedores de Immich quedaban en estado \texttt{healthy}. Esta operación no solo libera presión sobre el disco principal: también clarifica qué debe respaldarse, qué puede crecer y qué no debe mezclarse con configuraciones o bases de datos.

\texttt{immich\_postgres} almacena el estado persistente. Aunque PostgreSQL es una base ampliamente utilizada, en Immich aparece empaquetada con extensiones específicas para soportar capacidades de búsqueda y tratamiento de metadatos \citep{postgres15Docs,immichDockerInstall}. Por eso no se debe sustituir ni limpiar como si fuera una base genérica. Su volumen forma parte del núcleo de datos de la biblioteca.

\texttt{immich\_machine\_learning} es una pieza especialmente interesante para la memoria porque usa ROCm. A diferencia de Frigate, donde se decidió no usar la GPU por compatibilidad y saturación, Immich sí aprovecha la aceleración AMD para tareas de machine learning sobre imágenes \citep{immichRequirements,rocmDocs}. Esta diferencia demuestra que la política de GPU no es ``usar siempre'' o ``no usar nunca'', sino asignar aceleración donde aporta valor sin comprometer la estabilidad general.

\texttt{immich\_redis}, desplegado con Valkey, actúa como cola y caché de la aplicación \citep{valkeyDocs}. Su estado es menos crítico que la base de datos, pero sigue siendo necesario para que la aplicación funcione correctamente. En términos de arquitectura, Immich enseña una lección repetida en Odín: un servicio visible suele depender de varias piezas invisibles. Documentar solo \texttt{immich\_server} ocultaría la complejidad real del despliegue.

\subsection{Stack \texttt{audio}: STT, ASR y síntesis de voz}
La capa de audio reúne cuatro contenedores: Piper, faster-whisper, Wyoming Whisper y el ASR propio de Odín. Esta zona es una de las más experimentales del proyecto porque la interacción por voz exige latencia baja, naturalidad y compatibilidad con el hardware disponible. Además, al trabajar con GPU AMD, muchas soluciones orientadas a CUDA no son trasladables directamente.

\texttt{piper} se mantiene como TTS estable. No es necesariamente la voz más realista, pero cumple un requisito fundamental: responder rápido y de forma fiable \citep{linuxserverPiperDocs}. En un asistente físico o móvil, la respuesta inmediata puede ser más valiosa que una voz perfecta que llega demasiado tarde.

\texttt{faster-whisper-server} proporciona STT mediante una interfaz de servidor. La imagen desplegada es CPU, lo que reduce dependencia de ROCm y evita competir con Ollama o Immich por GPU. Esta elección sacrifica parte del rendimiento potencial, pero mejora aislamiento y previsibilidad. El proyecto \texttt{speaches} mantiene una aproximación compatible con APIs de voz, lo que facilita integraciones posteriores \citep{speachesDocs}.

\texttt{wyoming-whisper} existe por integración. El protocolo Wyoming se usa dentro del ecosistema de Home Assistant para conectar satélites de voz y servicios de transcripción; por eso esta pieza no se evalúa solo por precisión, sino por encaje con la domótica \citep{wyomingProtocolRepo,wyomingWhisperRepo,homeAssistantDocs}. En Odín, donde Home Assistant vive en una Raspberry Pi 5 y el servidor principal aporta cómputo, esta separación puede ser útil: HA coordina dispositivos y Odín ejecuta tareas más pesadas.

\texttt{odin-asr} es el servicio propio basado en ONNX. Su importancia académica es distinta a la de los contenedores anteriores: no se incluye solo para ``tener otro STT'', sino para experimentar con control local, formatos de modelo y despliegue propio. Aunque una herramienta externa pueda ser más madura, un componente propio permite justificar decisiones, medir límites y mostrar aportación técnica en la memoria.

\subsection{Stacks especializados: cámaras, Home Assistant, monitorización y TTS}
\texttt{frigate} cubre la videovigilancia. En Odín se usa para detección de personas y cámaras, pero se ha tomado una decisión concreta: usar CPU para detección en lugar de GPU. La razón es práctica. La GPU AMD ya se utiliza en Ollama e Immich, y en Frigate la compatibilidad y la saturación no compensaban la mejora esperada. Como la detección funciona suficientemente bien en CPU, se reserva la GPU para cargas donde sí aporta una diferencia clara \citep{frigateInstallDocs,rocmDocs}.

\texttt{ha-proxy} es deliberadamente simple. Home Assistant no se ha levantado en Docker en Odín porque ya está instalado en la Raspberry Pi 5. El contenedor basado en \texttt{alpine/socat} solo actúa como puente hacia esa instancia \citep{alpineSocatDockerHub,homeAssistantDocs}. Esta decisión evita duplicar HA, reduce riesgo y respeta la arquitectura real de la casa: la Raspberry Pi queda como centro domótico y Odín como servidor de IA, memoria y automatización.

\texttt{netdata} aporta la capa visual de monitorización. A diferencia del JSON de estado, pensado para sensores y automatizaciones, Netdata ofrece un panel navegable con CPU, memoria, red, discos, procesos, Docker y alarmas \citep{netdataDockerInstall}. Inicialmente se probó su integración mediante una tarjeta Webpage/iframe \citep{homeAssistantIframeCard}, pero la versión final utiliza tarjetas nativas y conserva Netdata como enlace opcional debido a restricciones de contenido mixto y acceso a red privada.

\texttt{pockettts-server} es el TTS experimental compatible con estilo OpenAI. Su papel actual es servir como línea de prueba para una voz más natural y conectable desde interfaces que ya entienden APIs de tipo OpenAI. No sustituye todavía a Piper, porque la experiencia de voz exige estabilidad y baja latencia. En la memoria debe aparecer como ejemplo de iteración técnica: se probaron varias alternativas, se mantuvo una opción estable y se dejó otra como línea de mejora.

\subsection{Criterios de operación y limpieza}
El inventario también permitió definir una política de limpieza. Se ejecutaron acciones seguras como eliminar contenedores detenidos, caché de construcción e imágenes no utilizadas. En cambio, se evitaron acciones destructivas como borrar volúmenes, modelos, bases de datos, la carpeta de cuarentena o el origen antiguo de Immich. Esta prudencia es necesaria porque en un sistema experimental los restos no son siempre basura: a veces son evidencias de decisiones, pruebas fallidas o datos que aún no han sido migrados.

El objetivo a medio plazo es separar \texttt{odin-master} en dominios más claros: núcleo conversacional, automatización, almacenamiento, ingesta y utilidades. La reorganización ya ha empezado con carpetas específicas dentro de \nolinkurl{/home/k1k3/odin}: \texttt{core}, \texttt{audio}, \texttt{media}, \texttt{automation}, \texttt{monitoring} y \texttt{scripts}. La regla será mover primero la documentación, después separar secretos a \texttt{.env}, luego dividir Compose sin tocar volúmenes y finalmente limpiar carpetas históricas con inventario previo.

\subsection{Flujo: guardar notas y documentos}
La herramienta \texttt{guardar\_nota} implementa uno de los requisitos del bloque ``Hacer'': guardar ideas, notas y archivos. Recibe tres parámetros: carpeta, título de archivo y contenido en Markdown. La descripción limita las carpetas permitidas, por ejemplo \texttt{Ideas Proyectos}, \texttt{Recetas}, \texttt{Viajes}, \texttt{Documentos Personales} u \texttt{Otros}. Esta restricción evita que el modelo cree rutas arbitrarias y reduce errores de organización.

El flujo es el siguiente. Primero, el usuario pide guardar una idea, resumen, receta o artículo. Después, el modelo estructura el contenido en Markdown y llama a \texttt{guardar\_nota\_nextcloud}. La tool envía un \texttt{POST} a un webhook de n8n con la carpeta, el nombre de archivo y el contenido. Finalmente, n8n escribe la nota en Nextcloud y la tool devuelve confirmación. Esta separación es buena arquitectura: Open WebUI interpreta la intención, n8n ejecuta la integración y Nextcloud almacena el resultado.

\subsection{Flujo: consultar memoria privada}
\texttt{buscar\_nextcloud} cubre la búsqueda en documentos, notas, apuntes y proyectos. Su propia descripción obliga al modelo a usarla cuando la pregunta pueda estar en información privada almacenada. Esto es una decisión importante: el asistente no debe responder de memoria genérica si existe una fuente personal potencialmente más fiable.

El flujo comienza con una consulta en lenguaje natural. La tool la envía a la capa de búsqueda asociada a Nextcloud y devuelve contexto textual. El modelo debe basar la respuesta en ese contexto, indicar si no hay resultados y no inventar contenido documental. En la memoria se puede presentar como una garantía de trazabilidad: Odín no solo ``recuerda'', sino que diferencia entre conocimiento general, documentos personales y ausencia de evidencia.

\subsection{Flujo: fotos y recuerdos visuales}
\texttt{buscar\_fotos} conecta el chat con Immich. Recibe una descripción en lenguaje natural, por ejemplo una persona, lugar, objeto, evento o escena, y devuelve una línea Markdown de imagen. La descripción de la herramienta incluye una instrucción crítica: el modelo debe copiar esa línea sin modificarla, porque el formato \texttt{![texto](url)} es lo que permite renderizar la foto directamente en el chat.

Este detalle parece pequeño, pero es importante para la experiencia de usuario. No basta con decir ``he encontrado una foto''; la utilidad real es que la imagen aparezca en la conversación. Esta tool convierte Immich en memoria visual accesible desde lenguaje natural, que era uno de los objetivos del mapa inicial.

\subsection{Flujo: web, vídeos e ingesta de conocimiento}
La herramienta \texttt{guardar\_transcripcion\_de\_video\_o\_pagina\_web} contiene dos funciones. \texttt{get\_youtube\_transcript} extrae texto de un vídeo de YouTube, mientras que \texttt{search\_and\_crawl\_web} puede leer una URL concreta o buscar contenido en internet mediante Crawl4AI. Internamente usa librerías como \texttt{youtube\_transcript\_api}, \texttt{crawl4ai}, \texttt{aiohttp}, \texttt{tiktoken} y estrategias de generación de Markdown.

Este flujo permite que Odín convierta contenido externo en material procesable. Un vídeo, artículo o página web deja de ser una referencia efímera y pasa a ser texto que puede resumirse, guardarse o incorporarse a la memoria. Esta capacidad encaja con el bloque ``Pensar'': navegar entre ideas, notas y archivos no se limita a buscar dentro de lo ya guardado, sino también a capturar nuevo conocimiento.

\subsection{Flujo: salud del sistema y autoreparabilidad}
\texttt{salud\_del\_sistema} ofrece la función \texttt{manage\_system\_health}. Acepta una acción y, opcionalmente, un nombre de servicio. Las acciones documentadas son \texttt{gpu\_status}, \texttt{cpu\_status}, \texttt{general}, \texttt{processes}, \texttt{clean} y \texttt{restart}. La tool envía la petición a un webhook de n8n, que actúa como frontera de ejecución.

La descripción impone una regla de seguridad: el modelo no debe inventar métricas, procesos ni servicios. Si Quique pide reiniciar algo, debe indicar el servicio concreto; si no lo indica, la herramienta debe usarse en modo diagnóstico. Este diseño reduce el riesgo de que una conversación provoque una acción destructiva por ambigüedad. La herramienta se complementa con \texttt{odin\_autorepair.py}, los informes de Telegram y el exportador de estado para Home Assistant.

\subsection{Flujo: cámaras y vigilancia}
La herramienta de vigilancia de Frigate integra las cámaras en la conversación. La función de consulta obtiene las cámaras conectadas; la función de foto actual devuelve la captura más reciente; la consulta del último evento recupera una detección grabada; y \texttt{ultima\_persona} filtra eventos de tipo persona, usa el reconocimiento disponible en \texttt{sub\_label} y devuelve identidad y captura.

Frigate detecta actualmente la cámara \texttt{tapo\_salon}, que en la interacción se considera la cámara de la entrada. Si solo existe una cámara, la tool la selecciona automáticamente. Las imágenes no se entregan directamente desde la API interna: un proxy mínimo en el puerto 2293 valida la petición, recupera el JPEG y añade cabeceras CORS. Igual que en Immich, las funciones visuales devuelven Markdown de imagen y el modelo debe conservarlo para que se renderice en el chat.

\subsection{Flujo: domótica y Home Assistant}
La integración final de Home Assistant utiliza una única tool asignada a Odín. Trabaja de forma estructurada: busca entidades, ejecuta acciones específicas, gestiona calendario, lista de la compra y datos de salud. El conector duplicado que delegaba lenguaje natural en Home Assistant Assist fue retirado del modelo porque podía elegir una ruta incorrecta o devolver una respuesta conversacional sin ejecutar la acción física.

La búsqueda de entidades se amplió con alias y dominios. Por ejemplo, \textit{aspirador}, \textit{robot}, \textit{Dreame} y \textit{D10} se asocian al dominio \texttt{vacuum}; luces a \texttt{light}; enchufes a \texttt{switch}; persianas a \texttt{cover}; y clima a \texttt{climate}. Esta corrección fue necesaria porque una comparación textual simple no relacionaba ``aspirador'' con la entidad \texttt{vacuum.d10\_plus\_gen\_2}.

Para el aspirador se aplicó además un patrón atómico. La función \texttt{controlar\_aspirador} recibe una acción natural y ejecuta directamente \texttt{vacuum.start}, \texttt{vacuum.pause}, \texttt{vacuum.stop} o \texttt{vacuum.return\_to\_base}. Después consulta el estado real antes de confirmar. Así se evita un fallo observado en modelos con tools: localizar correctamente una entidad, escribir ``ejecutando'' y no realizar la segunda llamada.

Además, \texttt{home\_assistant} incluye funciones para lista de la compra, calendario y salud. \texttt{manage\_shopping\_list} permite añadir, borrar o leer elementos. \texttt{manage\_calendar} crea eventos con fechas y horas estructuradas. \texttt{get\_health\_status} intenta leer datos de salud desde sensores expuestos en Home Assistant. Los temporizadores y la música no necesitan una tool propia porque se resuelven desde Home Assistant y HA Voice, que ya funcionan como capa domótica natural. Esto cubre varias ramas del mapa inicial: control domótico, recordatorios, lista de la compra, historial de salud y acciones desde conversación.

\subsection{Flujo: recetas con Mealie}
Las recetas se almacenaban inicialmente como notas Markdown en Nextcloud. Aunque el guardado funcionaba, ese formato no ofrecía una estructura específica para ingredientes, pasos, tiempos, raciones e imagen. Por este motivo se desplegó Mealie 1.12.0 como servicio local en el puerto 9925 y se añadió la tool \texttt{mealie\_recetas}.

La integración no expone directamente el token de Mealie. Un proxy local en el puerto 2292 ofrece operaciones limitadas para crear una receta desde texto, importar una URL compatible con el scraper de Mealie o importar una imagen. La carga de imágenes descarga el recurso público y lo adjunta a la receta mediante la API oficial. Las pruebas verificaron la creación, actualización de ingredientes y pasos, carga de imagen y eliminación posterior de las recetas de prueba. La interpretación de una imagen adjunta al chat depende de que el modelo seleccionado tenga capacidad de visión.

\subsection{Estabilizacion tras el cambio de red}
El traslado a una nueva red doméstica reveló una deuda de configuración: varias tools conservaban direcciones de la subred anterior \texttt{192.168.86.0/24}. La solución final evita sustituir una dependencia frágil por otra. Las comunicaciones entre servicios del mismo servidor usan \texttt{127.0.0.1}; los recursos que debe cargar el navegador se sirven mediante proxies LAN específicos.

Se añadieron cuatro proxies de propósito limitado: imágenes de Immich en el puerto 2290, notas de Nextcloud en 2291, operaciones de Mealie en 2292 e imágenes de Frigate en 2293. Los proxies separan secretos del código de las tools, reducen dependencia de webhooks públicos y permiten controlar tipo de contenido, CORS, caché y parámetros aceptados.

También se corrigió la interacción con YouTube. La librería \texttt{youtube-transcript-api} funcionaba correctamente, pero el modelo podía responder ``procesando'' sin llamar a la función. La descripción y el prompt obligan ahora a que la primera acción sea \texttt{get\_youtube\_transcript}, sin simular trabajo en segundo plano. Finalmente, se amplió el contexto de Odín de 4096 a 16384 tokens. El modelo admite un contexto mayor, pero 16384 equilibra conversaciones largas y consumo de memoria en un servidor con 30 GiB de RAM.

\subsection{Dashboard operativo de Home Assistant}
El cierre de la capa visual se realizó mediante un dashboard de Home Assistant llamado \textit{Odín}. El exportador \texttt{odin\_ha\_status.py} comprueba endpoints HTTP y TCP, resume Docker, discos, Qdrant y avisos de autoreparación y publica 24 entidades en Home Assistant mediante su API. El cron actualiza estos estados cada cinco minutos, por lo que el panel se recupera también después de reiniciar Home Assistant.

El dashboard contiene seis vistas: resumen operativo, catálogo de servicios, domótica y vigilancia, salud personal, organización cotidiana y monitorización. La vista de salud utiliza 25 entidades sincronizadas desde el Pixel 9 Pro mediante Health Connect y separa actividad, sueño, constantes cardiovasculares, respiración, oxígeno, composición corporal y métricas todavía sin lectura. La vista Organización incorpora las entidades nativas \texttt{todo.lista\_de\_la\_compra}, \texttt{todo.lista\_tareas\_casa} y el calendario. Las listas se pueden editar, completar y ampliar desde el propio panel.

La vista Monitor se rediseñó para evitar un iframe de Netdata. Cuando Home Assistant se abría desde una URL pública HTTPS, el navegador podía bloquear la conexión HTTP hacia una dirección privada o solicitar autenticación de nuevo. El panel utiliza ahora únicamente entidades nativas de Home Assistant para mostrar infraestructura, discos, servicios y avisos. Netdata queda como enlace opcional para diagnóstico profundo dentro de la LAN.

Las tarjetas de servicios muestran disponibilidad real y ofrecen acceso directo a Open WebUI, Nextcloud, Immich, Mealie, n8n, Qdrant, Frigate y Netdata. La configuración se conserva en \nolinkurl{home-assistant/odin-dashboard.json} y puede reinstalarse mediante \nolinkurl{tools/install_odin_dashboard.py}.

\subsubsection{Diseño visual y jerarquía de información}
La primera versión era funcional, pero se parecía demasiado a una colección de tarjetas independientes. La revisión visual tomó como referencia dashboards domésticos que priorizan una imagen ambiental, navegación estable y una cantidad reducida de información primaria \citep{redditWifeApprovedDashboardV2}. No se reprodujo su composición ni su identidad; se trasladaron tres principios: una portada reconocible, navegación por dominios y controles secundarios después del contexto.

La portada de Odín utiliza una imagen panorámica propia de un servidor doméstico integrado en un despacho nocturno. Sobre ella se muestran solo cuatro indicadores: estado global, contenedores, memoria vectorial y avisos. El resto se organiza debajo en bloques de exploración, estado operativo, salud, biblioteca personal y núcleo de inteligencia. Esta jerarquía permite comprender el sistema antes de interactuar con él.

La solución utiliza tarjetas nativas de Home Assistant para reducir dependencias. La cabecera se implementa con \texttt{picture-elements}; las métricas y acciones emplean \texttt{tile}, \texttt{button}, \texttt{gauge}, \texttt{history-graph}, \texttt{entities}, \texttt{todo-list}, \texttt{calendar}, \texttt{map}, \texttt{media-control}, \texttt{weather-forecast} y \texttt{markdown}. Resumen, Servicios, Salud, Organización y Monitor utilizan una rejilla de cuatro columnas; Casa conserva tres por la relación entre la cámara y el mapa del aspirador. Las cabeceras ocupan el ancho completo sin alterar el contenido ni el orden de las tarjetas editadas manualmente. La navegación y los accesos de Salud emplean el icono \texttt{mdi:heart-pulse}, de forma que el corazón identifica claramente este dominio.

La tipografía inicial de las cabeceras tenía poco peso y alcanzaba un tamaño excesivo en pantallas anchas. La revisión adopta la fuente principal de Home Assistant, peso 600 y un máximo de 40 píxeles. El resultado conserva el carácter editorial de la referencia, pero mejora legibilidad y coherencia con el resto de tarjetas.

La vista Casa se amplió con entidades ya disponibles en la instalación: Google TV, televisión del salón, Xbox Series X, Pixel 9 Pro, iPhone de Ivet y \texttt{weather.forecast\_casa}. Las tarjetas permiten controlar reproducción y volumen, consultar batería y dirección geocodificada, visualizar ambos móviles sobre el mapa y revisar la previsión de cinco días. Se descartaron reproductores antiguos en estado \texttt{unavailable} para que el panel represente el hogar operativo y no el historial completo de integraciones.

La primera versión servía la imagen desde el puerto 8765 de Odín. Este enfoque fallaba en algunos móviles y accesos remotos porque el navegador debía conectar desde una página pública a un recurso de la red privada. La versión definitiva sube las imágenes al almacenamiento gestionado por Home Assistant y utiliza rutas del mismo origen, \nolinkurl{/api/image/serve/.../original}. Así, la autenticación, el esquema HTTPS y la resolución de red son los mismos que los del dashboard.

Cada vista utiliza una imagen diferente y relacionada con su función: servidor doméstico para Resumen, racks para Servicios, interior residencial para Casa, tecnología de bienestar para Salud, lista escrita para Organización y estación de trabajo para Monitor. Las cabeceras externas proceden de Unsplash y se conservan localmente con atribución; la de Salud fue generada específicamente para Odín con un reloj inteligente y un pulsioxímetro en un entorno doméstico sereno. Todas fueron recortadas y comprimidas antes de subirlas a Home Assistant. De este modo, el panel no solicita recursos externos durante el uso.

Los dos enchufes Sonoff ya monitorizan la torre principal y la Raspberry Pi. Home Assistant registra potencia instantánea, tensión, corriente y energía diaria y mensual; también se habilitaron los contadores totales de energía que la integración había creado desactivados. La vista Monitor presenta controles de los enchufes, consumo actual, gráficas de potencia de 24 horas, evolución energética semanal y un cálculo de coste acumulado. Mientras no se introduzca la tarifa exacta del contrato se utiliza una estimación explícita de 0,20 euros por kWh. Las reservas se mantienen únicamente para el estado, fecha y tamaño del futuro backup NVMe.

\section{Problemas encontrados}
El principal problema técnico fue la combinación de IA local y GPU AMD. Aunque ROCm permite ejecutar cargas aceleradas, la compatibilidad práctica de algunas herramientas no fue directa. En TTS se probaron varias alternativas y la búsqueda de una voz realista y rápida se convirtió en uno de los mayores bloqueos. Piper quedó como solución funcional y HA Voice como capa práctica para interacción, mientras que opciones más ambiciosas quedaron limitadas por latencia, compatibilidad o madurez.

Otro problema importante fue la deuda operativa acumulada. Existían scripts, cron antiguos, contenedores históricos, rutas previas y servicios de prueba. Algunos seguían apuntando a ubicaciones anteriores o generaban errores aunque ya no fueran el camino principal. La solución no fue borrar masivamente, sino inventariar, mover con cuidado, mantener enlaces de compatibilidad cuando era necesario y desactivar lo obsoleto de forma reversible.

La gestión de almacenamiento también exigió intervención. Immich creció hasta ocupar una cantidad relevante de espacio, por lo que se migró su biblioteca a \nolinkurl{/mnt/almacen}. Esta operación obligó a detener únicamente el stack afectado, copiar datos, actualizar configuración, levantar servicios y validar healthchecks. El origen antiguo se conservó temporalmente como respaldo.

En seguridad, el reto ha sido equilibrar utilidad y exposición. Odín necesita hablar con muchos servicios: Home Assistant, n8n, Nextcloud, Immich, Frigate, Qdrant, Telegram y Open WebUI. Cada integración aumenta la superficie de riesgo. Por ello se evita publicar secretos en el repositorio, se documentan configuraciones saneadas, se usa Tailscale para acceso remoto y se plantea mover credenciales históricas a ficheros \texttt{.env}.
